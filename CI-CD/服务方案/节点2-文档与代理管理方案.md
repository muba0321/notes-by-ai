## 🔗 GitHub 联动方案

### GitHub 工作流

```
1. 节点 1 生成代码
   ↓
2. Git commit + push
   ↓
3. GitHub 仓库接收
   ↓
4. 触发 GitHub Actions（可选）
   ↓
5. 主节点 Pull 同步
   ↓
6. MkDocs 重新构建
```

### GitHub CLI 集成

**在所有节点配置：**

```bash
# 安装 GitHub CLI
apt-get install -y gh

# 配置 Token
gh auth login --with-token << 'EOF'
ghp_YOUR_GITHUB_TOKEN
EOF

# 验证
gh whoami  # 应返回 muba0321
```

## 📊 完整工作流示例

### 场景 1：新功能开发

```
1. 你在 Webchat 说：
   "开发一个用户管理 API，用 FastAPI"

2. 节点 2 (代理管理):
   → 分析任务类型（代码生成）
   → 分发到节点 1

3. 节点 1 (代码生成):
   → aliyun-coder 生成 FastAPI 代码
   → 创建项目结构
   → 安装依赖
   → 启动服务测试
   → Git commit + push

4. GitHub:
   → 接收代码
   → 触发 Actions（可选）

5. 主节点:
   → Pull 同步
   → 更新文档

6. 返回你：
   "用户管理 API 已部署，访问地址：http://..."
```

### 场景 2：文档更新

```
1. 你在 Webchat 说：
   "更新 K8s 部署文档"

2. 节点 2 (文档管理):
   → 读取现有文档
   → aliyun-coder 完善内容
   → Git commit + push

3. 主节点:
   → Pull 同步
   → MkDocs 重新构建

4. wiki.mubai.top:
   → 文档已更新
```

### 场景 3：Bug 修复

```
1. 你报告：
   "API 有 Bug，返回 500 错误"

2. 节点 2 (代理管理):
   → 分析错误日志
   → 分发到节点 1

3. 节点 1 (Bug 修复):
   → 读取错误日志
   → aliyun-coder 分析原因
   → 生成修复代码
   → 应用修复
   → 运行测试
   → Git commit + push

4. 返回你：
   "Bug 已修复，原因：XXX，修复方案：XXX"
```

## ⚠️ 注意事项

### 1. 数据安全

- ✅ 敏感信息不要提交到 Git
- ✅ API Key 使用环境变量
- ✅ 定期更新 SSH Key

### 2. 资源管理

| 节点 | 资源 | 建议 |
|------|------|------|
| 节点 1 | 2.3G 内存可用 | 避免同时运行多个大任务 |
| 节点 2 | 3.2G 内存可用 | 可运行监控服务 |

### 3. 网络延迟

- 节点 1→GitHub：海外直连，速度快
- 节点 2→GitHub：国内，可能需要代理
- 节点间同步：通过 Git，无实时要求
