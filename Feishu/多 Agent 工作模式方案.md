## 📖 文档说明

### 目的

1. **标准化** - 统一多 Agent 飞书集成的工作模式
2. **可追溯** - Agent 沟通留痕、存档审计
3. **高效协作** - Agent 之间、Agent 与人之间的高效配合
4. **安全可控** - 权限隔离、操作审计、风险管控
5. **可扩展** - 支持未来更多 Agent、更多渠道接入

### 评审流程

```
┌─────────────────────────────────────────────────────────┐
│                    方案评审流程                          │

1. 方案负责人填写文档

3. 用户确认（业务需求、资源审批）
         ↓
```

## 第二部分：工作流集成模式

### 2.1 现有工作流梳理

#### 工作流 A：CI/CD 部署流程

```
用户请求 → main Agent → 生成变更单 → 子节点 1 复核 → 用户确认 → 执行部署 → 归档
```

**飞书集成点：**
- ✅ 飞书消息通知（变更单创建、复核完成、部署完成）
- ✅ 飞书文档（变更单模板、部署方案）
- ✅ 飞书多维表格（任务队列、状态追踪）
- ✅ 飞书审批（用户确认环节）

#### 工作流 B：监控告警流程

```
监控告警 → monitor Agent → 初步分析 → 通知相关人员 → 处理 → 记录故障报告
```

**飞书集成点：**
- ✅ 飞书消息（告警通知）
- ✅ 飞书日历（值班安排）
- ✅ 飞书多维表格（故障记录）
- ✅ 飞书群组（应急沟通）

#### 工作流 C：产品开发流程

```
需求提出 → product Agent → 生成 PRD → 评审 → coding Agent → 代码生成 → 部署
```

**飞书集成点：**
- ✅ 飞书文档（PRD 文档）
- ✅ 飞书知识库（产品文档归档）
- ✅ 飞书多维表格（需求池、迭代计划）
- ✅ 飞书会议（评审会议）

### 2.2 多 Agent 协作模式

#### 模式 A：链式协作（推荐⭐）

```
用户 → main Agent → consult Agent → 返回结果 → main Agent → 用户
```

**场景：** 用户咨询业务问题，需要查询知识库

**流程：**
1. 用户向 main Agent 发送消息
2. main Agent 分析后转发给 consult Agent
3. consult Agent 查询知识库生成回复
4. main Agent 整合后回复用户
5. 飞书多维表格记录完整对话链

**实现方式：**
```bash
# main Agent 调用 consult Agent
sessions_send --label=consult --message="用户咨询：xxx"

# 等待回复后整合
# 记录到飞书多维表格
feishu_bitable_create_record --fields='{"对话链":"main→consult→main"}'
```

#### 模式 B：并行协作

```
用户 → main Agent ─┬─→ consult Agent → 结果 A ─┐
                   ├─→ coding Agent  → 结果 B ─┼→ 整合 → 用户
                   └─→ monitor Agent → 结果 C ─┘
```

**场景：** 复杂任务需要多个 Agent 并行处理

**流程：**
1. main Agent 接收任务
2. 并行调用多个 Agent
3. 等待所有结果
4. 整合后回复用户
5. 飞书多维表格记录各 Agent 贡献

#### 模式 C：接力协作

```
用户 → main Agent → coding Agent → product Agent → monitor Agent → 用户
```

**场景：** 完整的产品开发流程

**流程：**
1. main Agent 接收需求
2. 转发给 coding Agent 生成代码
3. 转发给 product Agent 生成文档
4. 转发给 monitor Agent 配置监控
5. 最终回复用户
6. 飞书多维表格记录完整流程

### 2.3 人机协作模式

#### 模式 A：人工审批节点

```
Agent 处理 → 飞书审批 → 人工确认 → Agent 继续执行
```

**实现：**
```
飞书审批 API → 创建审批单 → 推送给审批人 → 等待审批 → 回调通知 Agent
```

#### 模式 B：人工介入节点

```
Agent 处理 → 遇到不确定 → 飞书消息@人工 → 人工回复 → Agent 继续
```

**实现：**
```
feishu_chat --action=send --message="@用户 请确认：xxx"
# 等待用户回复后继续
```

| main | 完整权限 | 读写 | 只读 |
| consult | 文档读取 | 只读 | 无 |
| coding | 文档编辑 | 读写 | 只读 |
| monitor | 消息发送 | 只读 | 只读 |

### 6.2 操作审计

**审计内容：**
- ✅ 所有飞书消息记录
- ✅ 所有 Git 提交记录
- ✅ 所有 SSH 连接记录
- ✅ 所有 Skills 调用记录

**审计频率：**
- 每日：自动审查异常操作
- 每周：人工审查沟通记录
- 每月：完整审计报告

### 6.3 风险控制

| 风险 | 控制措施 |
|------|---------|
| 敏感信息泄露 | 飞书消息加密、Git 凭证脱敏 |
| 未授权操作 | 飞书审批节点、双重确认 |
| Skills 安全 | 仅安装通过安全审计的 Skills |
| Agent 失控 | 紧急停止开关、人工介入机制 |

## 第七部分：故障排查

### 7.1 常见问题

#### 问题 1：Agent 无法接收飞书消息

**检查：**
```bash
# 1. 插件状态
openclaw plugins list | grep feishu

# 2. Gateway 状态
openclaw gateway status

# 3. 通道状态
openclaw channels status --probe
```

**解决：**
```
飞书开放平台 → 事件订阅
→ 确认已订阅"接收消息"事件
→ 确认应用已发布
```

#### 问题 2：多维表格无法写入

**检查：**
```bash
feishu_app_scopes
```

**解决：**
```
飞书多维表格 → 权限
→ 添加飞书应用为协作者
→ 授予编辑权限
```

#### 问题 3：Agent 之间无法通信

**检查：**
```bash
sessions_list
sessions_send --label=xxx --message="test"
```

**解决：**
```
检查 openclaw.json 中的 agents.list 配置
确认所有 Agent 的 workspace 存在
重启 Gateway
```
